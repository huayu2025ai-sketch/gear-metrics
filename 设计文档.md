## 1. 项目概述

本系统是一个专为户外爱好者设计的装备资产管理与智能分析平台。通过结构化记录个人户外装备，对接大模型（LLM）实现自动化**气候穿搭组合推荐**与**理性消费新购审计**。

### 核心技术栈

- **框架:** Next.s 15 (App Router, Server Actions)
    
- **数据库 & 认证:** Supabase (PostgreSQL)
    
- **样式:** Tailwind CSS + shadcn/ui (暗黑主题)
    
- **AI 引擎:** Vercel AI SDK + Gemini 1.5 Pro / Claude 3.5 Sonnet
    

## 2. 数据库设计 (Supabase SQL)

在 Supabase SQL 执行器中运行以下脚本建表，确保字段对 AI 友好：

SQL

```
-- 创建户外装备表
create table public.outdoor_gears (
  id uuid default gen_random_uuid() primary key,
  user_id uuid references auth.users(id) on delete cascade, 
  name text not null,                                      -- 装备名称
  brand text not null,                                     -- 品牌
  category text not null check (category in ('长袖T恤', '短袖T恤', '棉服/抓绒', '羽绒服', '软壳/皮肤衣', '长裤', '短裤', '鞋履', '背包', '其他')), 
  color text,                                              -- 颜色
  size text,                                               -- 尺码
  fit_type text default '标准' check (fit_type in ('紧身', '标准', '宽松')), -- 版型（防层叠冲突）
  min_temp integer default 15,                             -- 极限适用低温 (℃)
  max_temp integer default 25,                             -- 极限适用高温 (℃)
  purchase_price numeric(10, 2),                           -- 购入价格
  purchase_date date,                                      -- 购入时间
  status text default '在用' check (status in ('在用', '在途', '闲置', '损耗')), 
  remarks text,                                            -- 备注 (面料科技/克重/蓬松度)
  created_at timestamp with time zone default timezone('utc'::text, now()) not null
);

-- 开启行级安全策略 (RLS)
alter table public.outdoor_gears enable row level security;

-- 录入/读取策略
create policy "Users can control their own gears."
  on public.outdoor_gears for all using (auth.uid() = user_id);
```

## 3. 核心功能实现逻辑

### 阶段一：装备基础维护 (MVP)

实现装备的增删改查（CRUD）。使用 Next.js Server Actions 直接与 Supabase 通信，放弃编写繁琐的 API 路由。

- **数据导入:** 支持 Notion 导出的 CSV 文件一键 Mapping 导入。
    
- **前端交互:** 提供极简单页，默认**暗黑模式**。包含表单快速录入、分类 Filter 过滤、状态切换标签。
    

### 阶段二：AI 智能引擎接入

利用大模型的长上下文能力，将结构化数据清洗后转为 Prompt 的上下文。

#### 1. 智能三层穿衣推荐

- **输入参数:** 目的地、活动类型（如徒步、露营、攀冰）、预计温标范围（$T_{min} \sim T_{max}$）。
    
- **数据流:** 1. 前端触发请求，Server 端从数据库捞取该用户所有 `status = '在用'` 的装备。
    
    2. 过滤掉不相关品类（如背包、鞋履），只保留衣服鞋袜。
    
    3. 组装 JSON 丢给大模型。
    
- **大模型约束:** 必须严格遵循户外三层穿衣理论（排汗、保暖、防风雨），若用户库中缺少某一层，必须直接指出“当前装备库无法覆盖该温标，缺少XX层”。
    

#### 2. 新购审计（反冲动消费）

- **输入参数:** 用户拟购买的装备名称、品牌、单价、功能。
    
- **数据流:** 1. AI 读取现有同品类及临近品类的装备资产。
    
    2. 计算查重率与功能冗余度。
    
- **大模型约束:** 独立思考，禁止盲目附和。从功能重叠、预算溢价率、系统互补性三个维度严厉审计，最终输出 1-10 的购买指数。

```
'use server';

import { supabase } from '@/lib/supabase';
import { revalidatePath } from 'next/cache';

export async function fetchGears() {
  const { data, error } = await supabase
    .from('outdoor_gears')
    .select('*')
    .order('created_at', { ascending: false });
  if (error) throw new Error(error.message);
  return data;
}

export async function createGear(formData: FormData) {
  const data = {
    name: formData.get('name') as string,
    brand: formData.get('brand') as string,
    category: formData.get('category') as string,
    min_temp: parseInt(formData.get('min_temp') as string) || 0,
    purchase_price: parseFloat(formData.get('purchase_price') as string) || 0,
    status: formData.get('status') as string,
  };

  const { error } = await supabase.from('outdoor_gears').insert([data]);
  if (error) return { success: false, error: error.message };

  revalidatePath('/gears');
  return { success: true };
}
```

## 5. 开发漏洞防御清单（独立审查）

1. **非标数据断层:** 大模型无法纯靠 `NJ2LQ05` 这种货号理解衣服厚度。**开发要求：** 录入端必须有 `min_temp`（最低适用温度）的硬性约束。
    
2. **大模型输出崩溃:** 大模型自由发挥的文本会导致 `JSON.parse` 报错。**开发要求：** 必须使用 Vercel AI SDK 的 `generateObject` 或大模型的 `responseSchema` 强制结构化输出，前端使用 `zod` 做数据兜底校验。
    
3. **Token 冗余:** 装备变多后全量发送会导致严重的上下文延迟和资费浪费。**开发要求：** 进阶阶段必须引入前置 Embedding 向量检索或基础 SQL 预筛选。

### 6.5 Supabase SQL v1（可执行）

```sql
-- 1) 建表（若不存在）
create table if not exists public.outdoor_gears (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  name text not null,
  brand text not null,
  category text not null check (
    category in ('长袖T恤', '短袖T恤', '棉服/抓绒', '羽绒服', '软壳/皮肤衣', '长裤', '短裤', '鞋履', '背包', '其他')
  ),
  color text,
  size text,
  fit_type text not null default '标准' check (fit_type in ('紧身', '标准', '宽松')),
  min_temp integer not null default 15,
  max_temp integer not null default 25,
  purchase_price numeric(10,2) check (purchase_price is null or purchase_price >= 0),
  purchase_date date,
  status text not null default '在用' check (status in ('在用', '在途', '闲置', '损耗')),
  remarks text,
  created_at timestamptz not null default timezone('utc'::text, now()),
  updated_at timestamptz not null default timezone('utc'::text, now()),
  constraint outdoor_gears_temp_range_chk check (min_temp <= max_temp)
);

-- 2) 索引
create index if not exists idx_outdoor_gears_user_created_at
  on public.outdoor_gears (user_id, created_at desc);

create index if not exists idx_outdoor_gears_user_category_status
  on public.outdoor_gears (user_id, category, status);

create index if not exists idx_outdoor_gears_user_status_temp
  on public.outdoor_gears (user_id, status, min_temp, max_temp);

-- 3) 自动更新时间戳
create or replace function public.set_outdoor_gears_updated_at()
returns trigger
language plpgsql
as $$
begin
  new.updated_at = timezone('utc'::text, now());
  return new;
end;
$$;

drop trigger if exists trg_outdoor_gears_set_updated_at on public.outdoor_gears;
create trigger trg_outdoor_gears_set_updated_at
before update on public.outdoor_gears
for each row execute function public.set_outdoor_gears_updated_at();

-- 4) RLS
alter table public.outdoor_gears enable row level security;

drop policy if exists "Users can view own gears" on public.outdoor_gears;
create policy "Users can view own gears"
on public.outdoor_gears
for select
using (auth.uid() = user_id);

drop policy if exists "Users can insert own gears" on public.outdoor_gears;
create policy "Users can insert own gears"
on public.outdoor_gears
for insert
with check (auth.uid() = user_id);

drop policy if exists "Users can update own gears" on public.outdoor_gears;
create policy "Users can update own gears"
on public.outdoor_gears
for update
using (auth.uid() = user_id)
with check (auth.uid() = user_id);

drop policy if exists "Users can delete own gears" on public.outdoor_gears;
create policy "Users can delete own gears"
on public.outdoor_gears
for delete
using (auth.uid() = user_id);
```
